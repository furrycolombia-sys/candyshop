`general-purpose`

## Description

Ensures React components follow project standards and patterns.

## Scope
Path: {scope_path}

## Rules Reference
Reference (DO NOT read this file - use the checklist below instead): .claude/rules/component-patterns.md

## Generated Code Policy
- Treat files as generated if the path includes `generated`/`__generated__` or the header says "auto-generated", "generated by", or "DO NOT EDIT".
- Do NOT suggest refactors, improvements, or cleanups for generated files.
- Only report consistency issues (typing, naming, signature/shape mismatches) that affect usage.

## KISS Guardrail
- Prefer the simplest fix that resolves the issue.
- Avoid abstractions or refactors that add complexity without clear payoff.
- If KISS conflicts with DRY or other guidance, note it and favor KISS.

## Context Management (CRITICAL)
- **Grep first, Read second**: Use Grep to find violation patterns BEFORE reading files. Only Read files that Grep identifies as having issues.
- **Never read generated files**: Skip any path containing `generated` or `__generated__` entirely.
- **File read limit**: Read at most 30 source files total. If more files need inspection, prioritize by: (1) feature code over shared, (2) .tsx over .ts, (3) larger files over smaller.
- **Use Read with limit**: For files over 200 lines, use the `limit` parameter to read only the relevant section (use Grep line numbers to target).
- **Don't read rule files**: The rules are already summarized in "What to Check" above. Do NOT Read the .claude/rules/*.md files - use the checklist provided instead.
- **Save early if needed**: If you've analyzed 25+ files, write your findings immediately even if you haven't covered everything. Partial results are better than no results.

## Already Covered by ESLint (DO NOT CHECK)
These are enforced by the linter — skip them entirely:
- **One component per file**: `react/no-multi-comp` catches multiple component exports per file
- **Hook rules**: `react-hooks/rules-of-hooks` enforces hook call rules
- **Effect dependencies**: `react-hooks/exhaustive-deps` catches missing/wrong deps
- **Nested component definitions**: `react/no-unstable-nested-components` catches components defined inside other components

## What to Check

1. **Component Structure**
   - Should follow: hooks → derived state → handlers → early returns → render
   - Props destructured at top
   - Single return statement (after early returns)

2. **'use client' Directive**
   - Present for components using: useState, useEffect, onClick, onChange
   - NOT present for server components
   - Placed at very top of file

3. **Props Interface**
   - Co-located with component (ComponentName.types.ts or inline)
   - Named as ComponentNameProps
   - Properly typed (no `any`)

4. **Server/Client Components**
   - Data fetching in server components
   - Interactivity in client components
   - Proper composition

5. **Container/Presenter Pattern**
   - Data fetching separated from UI
   - Containers handle logic
   - Presenters are pure UI

## Analysis Process (Grep-First Strategy)

1. **Survey structure** - Use Glob to find all .tsx files (don't read yet). Note file count.
2. **Grep for missing 'use client'** - Search for files using `useState|useEffect` without `'use client'`:
   - First find files with hooks: `useState|useEffect|useCallback`
   - Then check if those files have `'use client'` at top
3. **Grep for `any` type usage** - Search for `: any` or `as any` patterns
4. **Only Read files where Grep found potential issues** - Verify the violations
5. **Spot-check 3-5 representative components** for structure order (hooks → derived → handlers → render)
6. **Write findings to JSON** when done (or after reading 25 files)
