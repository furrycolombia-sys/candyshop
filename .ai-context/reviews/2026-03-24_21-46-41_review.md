### [BUG-001] Race condition: requestAnimationFrame-based mount guard may fire before hydration dispatch completes

- **File:** `apps/store/src/features/cart/application/CartContext.tsx`
- **Line:** 134
- **Description:** The mount guard uses `requestAnimationFrame` to set `mountedRef.current = true` after the hydration effect runs. However, `requestAnimationFrame` timing is not guaranteed relative to React's commit phase. If the HYDRATE dispatch triggers a synchronous re-render that fires the persist effect before the rAF callback, the guard works correctly. But if React batches the state update and the rAF fires before the next render cycle's effect, the persist effect could still skip a legitimate user-triggered update. In practice this is unlikely to cause visible bugs because the hydration effect and the rAF callback both run in the same task/microtask frame, but the approach is fragile.
- **Fix:** Consider using a `useRef(false)` that is set to `true` inside the hydration effect itself (after dispatch), or use a dedicated `hydrated` state flag that the persist effect checks. A simpler pattern: set `mountedRef.current = true` at the end of the hydration useEffect (after the dispatch), which guarantees ordering since effects run sequentially.
